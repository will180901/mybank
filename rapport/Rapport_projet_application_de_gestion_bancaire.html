Modèle Conceptuel de Données (MCD)

Le MCD représente la structure conceptuelle de la base de données sans considération d'implémentation.

// Modèle Conceptuel de Données (MCD) pour MyBank

Table Utilisateur {
  id int [pk]
  nom_complet varchar
  email varchar
  mot_de_passe varchar
  date_inscription datetime
}

Table Compte {
  id int [pk]
  numero_compte varchar
  type_compte varchar // 'courant' ou 'epargne'
  solde decimal
  id_utilisateur int
  date_creation datetime
  derniere_operation varchar
}

Ref: Compte.id_utilisateur > Utilisateur.id

Table Transaction {
  id int [pk]
  type_operation varchar // 'depot', 'retrait', 'virement'
  montant decimal
  id_compte_source int
  id_compte_destination int [null]
  date_operation datetime
  motif varchar
  solde_avant decimal
  solde_apres decimal
}

Ref: Transaction.id_compte_source > Compte.id
Ref: Transaction.id_compte_destination > Compte.id

Légende du MCD

Le Modèle Conceptuel de Données représente les entités principales du système bancaire :

  • Utilisateur : Représente les clients de la banque
  • Compte : Décrit les comptes bancaires (courant ou épargne)
  • Transaction : Enregistre toutes les opérations financières

Les relations entre ces entités sont représentées par des flèches indiquant les clés étrangères.

Modèle Physique de Données (MPD)

Le MPD est la traduction technique du MCD avec les types de données précis et les contraintes.

Table Utilisateur

  • id : INTEGER (clé primaire, auto-incrément)
  • nom_complet : VARCHAR(100) (non null)
  • email : VARCHAR(100) (unique, non null)
  • mot_de_passe : VARCHAR(255) (non null, chiffré)
  • date_inscription : DATETIME (non null)

Table Compte

  • id : INTEGER (clé primaire, auto-incrément)
  • numero_compte : VARCHAR(20) (unique, non null)
  • type_compte : VARCHAR(10) ('courant' ou 'epargne')
  • solde : DECIMAL(15,2) (défaut 0.00)
  • id_utilisateur : INTEGER (clé étrangère vers Utilisateur.id)
  • date_creation : DATETIME (non null)
  • derniere_operation : VARCHAR(255)
  • decouvert_autorise : DECIMAL(15,2) (pour comptes courants)
  • taux_interet : DECIMAL(5,2) (pour comptes épargne)

Table Transaction

  • id : INTEGER (clé primaire, auto-incrément)
  • type_operation : VARCHAR(10) ('depot', 'retrait', 'virement')
  • montant : DECIMAL(15,2) (non null)
  • id_compte_source : INTEGER (clé étrangère vers Compte.id)
  • id_compte_destination : INTEGER (clé étrangère vers Compte.id, nullable)
  • date_operation : DATETIME (non null)
  • motif : VARCHAR(255)
  • solde_avant : DECIMAL(15,2)
  • solde_apres : DECIMAL(15,2)

Modèle Logique de Données (MLD)

Le MLD représente l'organisation des données dans le système de gestion de base de données.

Schéma relationnel

UTILISATEUR (id, nom_complet, email, mot_de_passe, date_inscription)

COMPTE (id, numero_compte, type_compte, solde, id_utilisateur, date_creation, derniere_operation, decouvert_autorise, taux_interet)

TRANSACTION (id, type_operation, montant, id_compte_source, id_compte_destination, date_operation, motif, solde_avant, solde_apres)

Dépendances fonctionnelles

  • id → (nom_complet, email, mot_de_passe, date_inscription) [UTILISATEUR]
  • id → (numero_compte, type_compte, solde, id_utilisateur, date_creation, derniere_operation, decouvert_autorise, taux_interet) [COMPTE]
  • id → (type_operation, montant, id_compte_source, id_compte_destination, date_operation, motif, solde_avant, solde_apres) [TRANSACTION]
  • numero_compte → (id, type_compte, solde, id_utilisateur) [COMPTE]

Modèle Conceptuel de Traitement (MCT)

Le MCT décrit les processus métier de l'application bancaire.

Processus : Création de compte

  • Acteur : Utilisateur
  • Préconditions : Utilisateur authentifié
  • Étapes :
    1. Sélection du type de compte (courant/épargne)
    2. Génération du numéro de compte
    3. Initialisation du solde à 0
    4. Enregistrement en base de données
    5. Notification de création réussie
  • Postconditions : Nouveau compte créé et visible dans l'interface

Processus : Effectuer une transaction

  • Acteur : Utilisateur
  • Préconditions : Compte existant avec solde suffisant
  • Étapes :
    1. Sélection du type de transaction (dépôt, retrait, virement)
    2. Saisie des détails (montant, compte cible, motif)
    3. Validation des contraintes métier
    4. Exécution atomique de l'opération
    5. Mise à jour des soldes concernés
    6. Enregistrement de la transaction
    7. Notification de succès
  • Postconditions : Soldes mis à jour, transaction enregistrée

Processus : Consultation de l'historique

  • Acteur : Utilisateur
  • Préconditions : Compte existant
  • Étapes :
    1. Sélection du compte à consulter
    2. Récupération des transactions associées
    3. Filtrage éventuel (date, type, montant)
    4. Présentation dans un tableau interactif
    5. Export possible (PDF, CSV)
  • Postconditions : Historique affiché à l'utilisateur
